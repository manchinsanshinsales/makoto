### 2026-06-13 Q. Bigtable行キー設計とホットスポット回避
- **ドメイン**: 3. データの保存
- **結果**: 正解
- **問題文**: 大規模IoTデバイスのセンサーデータをCloud Bigtableに保存する際、ホットスポットを防ぎつつ、特定デバイスの最新データをすばやく取得するための行キー（Row Key）の設計として最適なものを選ぶ。
- **復習メモ**: 行キーの先頭に十分に分散された値（device_id）を配置してホットスポットを回避し、後半に逆順タイムスタンプ（reverse_timestamp）を結合することで、特定デバイスの最新データを効率的にスキャン・取得できる。

### 2026-06-14 Q. Dataflowウィンドウ処理と遅延データ処理
- **ドメイン**: 2. データの取り込みと処理
- **結果**: 正解
- **問題文**: ゲームアプリのリアルタイムスコア集計において、重なりのない1時間単位で集計し、最大3時間の遅延データを許容しつつ、遅延データ到着時に即座に結果を更新出力する設計を選ぶ。
- **復習メモ**: 重なりのない固定時間集計にはタンブリングウィンドウを用い、遅延データ許容には `withAllowedLateness` を設定する。さらに遅延データ到着時に即時出力するためには、適切なトリガー（Triggering）の設定が必要である。

### 2026-06-14 Q. BigLakeとAnalytics Hubによる安全なゼロコピーデータ共有
- **ドメイン**: 1. データ処理システムの設計
- **結果**: 正解
- **問題文**: Cloud Storage上のデータを物理的にコピーせず、列レベルのセキュリティを維持したまま外部パートナーと共有する構成を選ぶ。
- **復習メモ**: 外部ストレージのデータにセキュリティ制限をかけるには BigLake を使用し、列レベルセキュリティにはポリシータグ（Policy Tags）を設定する。また、データを複製せずに組織内外に安全に共有するには Analytics Hub を使用するのがGoogle Cloudのベストプラクティス。

### 2026-06-14 Q. Cloud ComposerとWorkflowsの使い分け
- **ドメイン**: 5. データワークロードの保守と自動化
- **結果**: 正解
- **問題文**: DataprocやBigQueryなどの複数のタスク依存関係があり、数時間かかるバッチデータパイプラインの管理と、実行状況の詳細な可視化・失敗タスクの手動再実行を実現するオーケストレーションツールを選ぶ。
- **復習メモ**: 複雑な依存関係やDAGの可視化、詳細な監視、失敗した特定タスクからのリスタートが必要なバッチパイプラインには Cloud Composer が最適。一方、軽量・サーバーレスでHTTP APIのオーケストレーションやミリ秒起動が求められる処理には Workflows を選ぶ。

### 2026-06-14 Q. Cloud Natural Language API の機能の使い分け（エンティティ分析 vs 感情分析）
- **ドメイン**: 4. 分析のためのデータ準備と活用（AI/ML）
- **結果**: 不正解
- **問題文**: 機械学習の経験者や開発リソースが不足している状況下で、ユーザーのブログ投稿から自動的に「主題ラベル」を迅速に生成する方法を選ぶ。
- **復習メモ**: テキスト内のキーワードや固有名詞（主題ラベル）を抽出するには、人物・場所・物などを識別する「エンティティ分析」を使用する。「感情分析」はポジティブ/ネガティブといった感情トーンを識別する機能であるため主題抽出には適さない。

### 2026-06-14 Q. BigQuery外部テーブル（永続リンクテーブル）と圧縮Avroによる並列クエリ
- **ドメイン**: 3. データの保存 / 2. データの取り込みと処理
- **結果**: 不正解
- **問題文**: 非常に大きなテキストファイルのストレージ設計において、ANSI SQLによるクエリ、入力場所（GCS）からのデータ圧縮、および並列読み取り（ロード）をサポートする最適な構成を選ぶ。
- **復習メモ**: データをBigQueryに物理インポート（ロード）せず、安価なGCSに置いたまま高速クエリを行うには「永続リンクテーブル（外部テーブル）」が最適。また、圧縮されていても並列にスプリットして高速ロード・読み取りできるフォーマットとして「圧縮されたAvro」を選択するのがベストプラクティス。

### 2026-06-14 Q. BigQuery Data Transfer Service による YouTube データ転送と分析
- **ドメイン**: 2. データの取り込みと処理 / 3. データの保存
- **結果**: 不正解
- **問題文**: YouTubeチャンネルのログデータを分析のためにGoogle Cloudに転送し、世界のマーケティングチームが最新データに対してANSI SQLなどの分析を実行できる環境を構築する。
- **復習メモ**: YouTubeやGoogle広告などのGoogleのSaaSデータは「BigQuery Data Transfer Service (DTS)」を使ってBigQueryのデータセットへ直接自動転送するのが最も簡単で運用の手間がない（標準機能）。Storage Transfer Service (STS)は、異なるクラウドストレージ間やオンプレミスからGCSへのファイル転送に使用するサービスなので誤り。

### 2026-06-14 Q. Cloud Pub/SubからBigQueryへのストリーミング変換パイプラインの設計（Dataflow vs Dataproc）
- **ドメイン**: 2. データの取り込みと処理 / 5. データワークロードの保守と自動化
- **結果**: 不正解
- **問題文**: Pub/SubからBigQueryへJSONメッセージを書き込み変換する際、サービスコストを最小限に抑え、トラフィック変動があっても最小限の手動介入（自動化）で監視・対応できる構成を選ぶ。
- **復習メモ**: ストリーミング処理の負荷変動に対し、手動介入なしでコストを最適化するには、自動スケーリングを備えたフルマネージドな「Cloud Dataflow」が最適。Dataprocで手動スケーリング（ノード数調整）を行う構成は運用負荷が高いため不適切。また、ストリーミングジョブの監視指標には「総実行時間」ではなく、データの遅れを示す「システム遅延（System Lag）」を監視するのがベストプラクティス。

### 2026-06-14 Q. Cloud Video Intelligence API と Cloud Bigtable によるレコメンデーションエンジンの設計
- **ドメイン**: 4. 分析のためのデータ準備と活用（AI/ML） / 3. データの保存
- **結果**: 正解
- **問題文**: 過去の視聴履歴に基づく動画レコメンデーションにおいて、動画内のエンティティからラベルを生成し、数TB規模のデータに対して非常に高速なフィルタリング提案を行う構成を選ぶ。
- **復習メモ**: 動画内の物体・アクション・概念などを認識して自動タグ付けするには、学習済みAIの「Cloud Video Intelligence API」が最適（モデル自作は不要）。また、数TB以上の大量データに対してミリ秒単位の極めて低いレイテンシで読み書き（高速フィルタリング）できるデータベースには「Cloud Bigtable」がベスト。
