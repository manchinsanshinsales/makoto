### 2026-06-13 Q. Bigtable行キー設計とホットスポット回避
- **ドメイン**: 3. データの保存
- **結果**: 正解
- **問題文**: 大規模IoTデバイスのセンサーデータをCloud Bigtableに保存する際、ホットスポットを防ぎつつ、特定デバイスの最新データをすばやく取得するための行キー（Row Key）の設計として最適なものを選ぶ。
- **復習メモ**: 行キーの先頭に十分に分散された値（device_id）を配置してホットスポットを回避し、後半に逆順タイムスタンプ（reverse_timestamp）を結合することで、特定デバイスの最新データを効率的にスキャン・取得できる。

### 2026-06-14 Q. Dataflowウィンドウ処理と遅延データ処理
- **ドメイン**: 2. データの取り込みと処理
- **結果**: 正解
- **問題文**: ゲームアプリのリアルタイムスコア集計において、重なりのない1時間単位で集計し、最大3時間の遅延データを許容しつつ、遅延データ到着時に即座に結果を更新出力する設計を選ぶ。
- **復習メモ**: 重なりのない固定時間集計にはタンブリングウィンドウを用い、遅延データ許容には `withAllowedLateness` を設定する。さらに遅延データ到着時に即時出力するためには、適切なトリガー（Triggering）の設定が必要である。

### 2026-06-14 Q. BigQuery Data Transfer Service による YouTube データ転送と分析

- **結果**: 不正解
- **問題文**: YouTubeチャンネルのログデータを分析のためにGoogle Cloudに転送し、世界のマーケティングチームが最新データに対してANSI SQLなどの分析を実行できる環境を構築する。
- **復習メモ**: YouTubeやGoogle広告などのGoogleのSaaSデータは「BigQuery Data Transfer Service (DTS)」を使ってBigQueryのデータセットへ直接自動転送するのが最も簡単で運用の手間がない（標準機能）。Storage Transfer Service (STS)は、異なるクラウドストレージ間やオンプレミスからGCSへのファイル転送に使用するサービスなので誤り。

### 2026-06-14 Q. Cloud Pub/SubからBigQueryへのストリーミング変換パイプラインの設計（Dataflow vs Dataproc）
- **ドメイン**: 2. データの取り込みと処理 / 5. データワークロードの保守と自動化
- **結果**: 不正解
- **問題文**: Pub/SubからBigQueryへJSONメッセージを書き込み変換する際、サービスコストを最小限に抑え、トラフィック変動があっても最小限の手動介入（自動化）で監視・対応できる構成を選ぶ。
- **復習メモ**: ストリーミング処理の負荷変動に対し、手動介入なしでコストを最適化するには、自動スケーリングを備えたフルマネージドな「Cloud Dataflow」が最適。Dataprocで手動スケーリング（ノード数調整）を行う構成は運用負荷が高いため不適切。また、ストリーミングジョブの監視指標には「総実行時間」ではなく、データの遅れを示す「システム遅延（System Lag）」を監視するのがベストプラクティス。

### 2026-06-14 Q. Cloud Storage 暗号鍵漏洩とCMEK保護強制
- **ドメイン**: 1. データ処理システムの設計 / 3. データの保存
- **結果**: 演習
- **問題文**: Cloud KMSのCMEK鍵漏洩時におけるGCSデータの再暗号化と古い鍵の削除、および今後のCMEK未適用オブジェクトの書き込み防止策。
- **復習メモ**: 既存オブジェクトは新しい鍵を指定して `gsutil rewrite`（再暗号化）した後に古い鍵を破棄する。新規書き込みのCMEK強制には、バケットの「デフォルトKMS鍵」の設定や、組織ポリシーによるCMEK強制（restrictNonCmekServices）を設定する。

### 2026-06-14 Q. Cloud SQL への安全なパブリックIP接続とAuth Proxy
- **ドメイン**: 1. データ処理システムの設計 / 3. データの保存
- **結果**: 演習
- **問題文**: 動的パブリックIPを持つアプリ群から、SSL必須のCloud SQLへパブリックIP経由で安全に接続する構成を選ぶ。
- **復習メモ**: 「承認済みネットワーク」を空のままにし、すべてのアプリで「Cloud SQL Auth Proxy」を使用するのが最適。これにより、0.0.0.0/0 のような危険な開放をせずに、IAMによる強力な認証と暗号化された安全な接続が可能になる。

### 2026-06-14 Q. VPC Service Controls による特定プロジェクト内のPub/Subトピック保護
- **ドメイン**: 1. データ処理システムの設計
- **結果**: 演習
- **問題文**: プロジェクトAのPub/Subトピックのデータに対して、他のプロジェクトや外部からのアクセスを完全に禁止し、プロジェクトA内のみでアクセスを完結させる。
- **復習メモ**: 組織で「VPC Service Controls (VPC-SC)」を構成し、「プロジェクトAを囲むサービス境界」を設定する。これにより、IAM権限の有無にかかわらず、境界を越えたマネージドサービス（Pub/Subなど）へのアクセスをブロックできる。

### 2026-06-14 Q. BigQueryでのオンプレミスHSM管理鍵による暗号化（Cloud EKM）
- **ドメイン**: 1. データ処理システムの設計 / 3. データの保存
- **結果**: 演習
- **問題文**: BigQueryのデータを暗号化する際、暗号鍵マテリアルをGoogle Cloud内に保存せず「オンプレミスのHSM内でのみ生成・保存」して使用する。
- **復習メモ**: 鍵の生成・保管を完全にオンプレミス側で行う要件には「Cloud External Key Manager (Cloud EKM)」を使用してCloud KMSとリンクさせる。鍵をGoogle Cloud内にインポート（KMS/Cloud HSM）する方式では、Google側に鍵が保管されてしまうため不適切。

### 2026-06-14 Q. Cloud Storage のデュアルリージョンとターボレプリケーションによる RPO=15分と低遅延
- **ドメイン**: 1. データ処理システムの設計 / 3. データの保存
- **結果**: 演習
- **問題文**: 米国内でデータ損失を15分以下（RPO=15分以下）に抑え、リージョン障害時の復旧を行いつつ、us-central1内のDataproc（Spark）からの読み取りレイテンシを最小にする設計を選ぶ。
- **復習メモ**: デュアルリージョンバケット（us-central1とus-south1）を選択し、かつ「ターボレプリケーション」を有効にすることで、15分以内のデータ複製（RPO=15分以下）がSLAで保証される。また、Dataprocクラスタがバケットと同じリージョン（us-central1）からローカル読み取りするため、読み取りレイテンシを最小化できる。

### 2026-06-14 Q. Dataflow の Fusion（融解）回避と Reshuffle オペレーションによるボトルネック特定

### 2026-06-16 Q. BigQuery ML モデルのエクスポートと Vertex AI Endpoint による低レイテンシ推論
- **ドメイン**: 4. 分析のためのデータ準備と活用（AI/ML）
- **結果**: 正解
- **問題文**: BigQuery MLで学習させた予測モデルを使用して、本番Webアプリから10ミリ秒未満の超低レイテンシかつ高スループット（秒間数千リクエスト）でリアルタイム予測結果を返す設計を選ぶ。

### 2026-06-19 Q. Compute Engine（永続ディスク）のCMEKによる保存時暗号化
- **ドメイン**: 1. データ処理システムの設計 / 3. データの保存
- **結果**: 不正解
- **問題文**: Compute Engine上で実行されているKafkaおよびRedisクラスタの永続データを、作成・ローテーション・破棄が可能な暗号鍵で保護（保存時暗号化）する設計。
- **復習メモ**: 仮想マシン（Compute Engine）の保存データを暗号化するには、Cloud KMSで作成した暗号鍵（CMEK）を「インスタンス（永続ディスク）の暗号化キー」として適用する。アプリケーションのAPI呼び出しで都度KMSキーを参照して暗号化・復号をするアプローチはインフラレベルの透過的暗号化ではないため不適切。
